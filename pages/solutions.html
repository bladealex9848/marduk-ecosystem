Descripción general

El Sistema de Gestión de Expedientes Electrónicos (SGEJ) representa la columna vertebral de la transformación digital del sistema judicial colombiano. Esta plataforma integral permite el manejo completamente digital de los expedientes, desde su creación hasta su archivo definitivo.

Nivel 5: Consolidada
Expedientes Digital Búsqueda

Beneficios clave

  • Reducción del 40%

    en tiempos de gestión procesal

  • Disminución del 95%

    en el uso de papel

  • Acceso remoto y seguro

    a expedientes completos

Requisitos

  • Navegador web actualizado
  • Certificado digital para firma electrónica
  • Conexión a internet estable

Desarrollo de código

Cómo contribuir con código

Contribuye con nuevas funcionalidades, correcciones de errores o mejoras de rendimiento. Aceptamos pull requests en nuestro repositorio de GitHub. Nuestro código sigue estándares de calidad y pasa por revisiones exhaustivas.

Pasos para contribuir:
  1. Revisa los issues abiertos o crea uno nuevo
  2. Haz fork del repositorio
  3. Crea una rama para tu contribución
  4. Implementa tus cambios siguiendo las guías de estilo
  5. Envía un pull request

Proceso de contribución

1

Selecciona un issue o propuesta

Revisa los issues abiertos en GitHub o propone una nueva funcionalidad. Comenta en el issue para indicar que estás trabajando en él.

2

Haz un fork del repositorio

Crea un fork del repositorio en tu cuenta de GitHub y clona el proyecto en tu máquina local para comenzar a trabajar.

3

Desarrolla tu contribución

Trabaja en tu contribución siguiendo las guías de estilo y mejores prácticas del proyecto. Asegúrate de incluir pruebas si es necesario.

4

Envía un Pull Request

Una vez que hayas completado tu contribución, envía un Pull Request al repositorio principal con una descripción clara de los cambios.

Guías de contribución

Todo el código debe seguir las guías de estilo establecidas. Utilizamos ESLint para JavaScript, Checkstyle para Java y Black para Python. Asegúrate de que tu código pase todas las verificaciones de estilo antes de enviar un Pull Request.

Todas las contribuciones deben incluir pruebas adecuadas. Mantenemos un mínimo de 80% de cobertura de código. Las pruebas unitarias deben ser independientes y no depender de servicios externos o del estado de la aplicación.

Cualquier nueva funcionalidad o cambio significativo debe estar documentado. Esto incluye comentarios en el código, actualización de la documentación técnica y, si es necesario, actualización de los manuales de usuario.
